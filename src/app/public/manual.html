Iridian Lexicon Platform User Manual — v0.1

Platform for the Iridian constructed language · YAML canonical source · SQLite projection · Node/TypeScript

Overview

The Iridian Lexicon Platform is a self-contained system for maintaining the canonical lexical and grammatical data for the Iridian constructed language. It has three primary purposes:

  • Canonical data store — YAML files (one per entry) are the authoritative source of truth for all lexemes, morphemes, and rules.
  • Fast local index — A SQLite database is generated from the YAML files and used by the editing app for search and display. It is a derived projection, not a source of truth, and can be rebuilt at any time.
  • Grammar repo integration — JSON and LaTeX artifacts are generated into a staging folder (TEMP/) and consumed by the reference grammar at compile time.
Key principle
SQLite is never edited directly. All data flows YAML → pipeline → SQLite. If the DB is lost or corrupted, run lexicon rebuild to restore it from YAML.

Data Model

Lexemes

A lexeme is a full vocabulary item — a noun, verb, modifier, or function word. Every lexeme file must include:

FieldTypeNotes
schema_versionstringCurrent: "1.0"
idstringStable machine UUID-like identifier. Never changes.
keystringHuman-readable stable key used in YAML cross-refs and LaTeX macros.
lemmastringCitation form (plain text).
display_lemmastringDisplay form with diacritics.
entry_kind"lexeme"Discriminant field.
major_categoryenumnoun · verb · modifier · function-word
subtypestringE.g. "common-noun", "agentive-verb".
glossesstring[]English glosses; first entry is primary.
notesstringFree-text etymology/usage notes.
tagsstring[]Searchable tags.
statusenumactive draft deprecated
template_idstring?Key of the inflection rule to use. Either this or inflection_profile is required.
stem_variantsStemVariant[]Named stem forms consumed by the template.
manual_overridesobjectSlot → surface form; replaces generated forms.
usage_examplesUsageExample[]Structured token-based examples.
attested_instring[]Source references.

Morphemes

A morpheme is a bound or free grammatical element — a prefix, suffix, infix, circumfix, or particle. In addition to the core fields shared with lexemes, morphemes add:

FieldNotes
display_formThe canonical interlinear form, e.g. -ra, aN-.
gloss_abbrLeipzig-style abbreviation, e.g. NOM, PST.
slotThe morphological slot this morpheme occupies.
categoryParadigm dimension, e.g. case-system, TAM.
allomorph_rulesOrdered list of context-conditioned form alternations.

Rules

There are three rule layers:

  • Inflection rules (rules/inflection/) — paradigm templates. A lexeme declares template_id: noun-class-a and the engine generates all inflected forms by applying the template to the lexeme's stems.
  • Morphotactic rules (rules/morphotactics/) — word-building templates: which slots exist, their linear order, and co-occurrence constraints.
  • Syntax rules (rules/syntax/) — reusable clause templates. They validate structured token sequences without parsing raw strings.

Usage Examples

Examples are authored as structured token sequences, not freeform DSL strings. Each token explicitly references a lexeme form, a morpheme, or carries literal text. This makes examples resilient to future grammar changes.

usage_examples:
  - id: "ex-ara-001"
    source_lang:
      - type: form
        key: ara
        slot: NOM.SG
      - type: morpheme
        key: PST
      - type: form
        key: keth-run
        slot: ACC.SG
    gloss_line:
      - type: gloss
        gloss: "water"
      - type: gloss
        gloss: "PST"
      - type: gloss
        gloss: "river-ACC.SG"
    translation: "The water flowed in the river."

YAML File Format

Lexeme file

schema_version: "1.0"
id: "ara-001"
key: "ara"
lemma: "ara"
display_lemma: "āra"
entry_kind: lexeme
major_category: noun
subtype: common-noun
glosses:
  - "water"
  - "liquid"
notes: "Primary term for water. Used in compounds for liquid substances."
tags:
  - nature
  - body
status: active
template_id: noun-class-a
stem_variants:
  - label: base
    form: "ar"
  - label: weak
    form: "are"
manual_overrides:
  GEN.PL: "aranthi"   # irregular genitive plural
usage_examples: []
attested_in:
  - "corpus-texts-vol1"

Morpheme file

schema_version: "1.0"
id: "nom-sg-001"
key: "NOM.SG"
lemma: "-a"
display_lemma: "-a"
entry_kind: morpheme
major_category: suffix
subtype: case-suffix
glosses:
  - "nominative singular"
notes: "Default case for subject nominals."
tags: []
status: active
display_form: "-a"
gloss_abbr: "NOM.SG"
slot: case
category: case-system
allomorph_rules:
  - context: "stem ends in vowel"
    form: "-ra"

Inflection rule

schema_version: "1.0"
id: "noun-class-a-001"
key: "noun-class-a"
name: "Noun Class A"
description: "Primary nominal declension for common nouns."
category: noun
feature_axes:
  - ["NOM", "ACC", "DAT", "GEN"]
  - ["SG", "PL"]
cells:
  - slot: "NOM.SG"
    features:
      - { feature: case, value: NOM }
      - { feature: number, value: SG }
    suffix: "-a"
  - slot: "ACC.SG"
    features:
      - { feature: case, value: ACC }
      - { feature: number, value: SG }
    suffix: "-an"
  # … more cells …
tags: []
notes: ""

Morphotactic rule

schema_version: "1.0"
id: "nominal-template-001"
key: "nominal-slots"
name: "Nominal word template"
description: "Slot ordering for nominal word forms."
word_class: noun
slots:
  - name: stem
    position: 0
    required: true
  - name: case
    position: 1
    required: true
  - name: number
    position: 2
    required: false
alternations:
  - name: vowel-harmony
    context: "stem ends in back vowel"
    result: "back-vowel allomorphs apply"
tags: []
notes: ""

Syntax rule

schema_version: "1.0"
id: "basic-sov-001"
key: "basic-sov"
name: "Basic SOV clause"
description: "Subject–Object–Verb order for transitive clauses."
word_order: SOV
strict_order: true
constraints:
  - position: 0
    required_type: form
    required_slot: "NOM.SG"
    label: subject
  - position: 1
    required_type: form
    required_slot: "ACC.SG"
    label: object
  - position: 2
    required_type: form
    label: verb
tags: []
notes: ""

The Import Pipeline

Every write operation — whether from the CLI or the web app — passes through this pipeline before SQLite is touched:

  1. Read — Discover all .yaml files in source directories, parse them.
  2. Schema version check — Reject files with unsupported versions before any further processing.
  3. In-memory migration — Files at older (but migratable) schema versions are upgraded in memory without touching disk.
  4. Zod validation — Every file is validated against its strict Zod schema. Any error halts the pipeline.
  5. Cross-file validation — Checks for duplicate id, duplicate key, and unknown template_id references.
  6. Normalize — Convert validated data to the internal canonical model with provenance fields.
  7. Rule engine — Generate all paradigm forms for every lexeme.
  8. SQLite rebuild — Drop all tables and rebuild from scratch. The DB is only written if all earlier steps succeed.
Important
SQLite is never modified if any validation step fails. This ensures the DB always reflects a valid state of the YAML source.

CLI Reference

Run CLI commands from the repo root:

npx tsx src/cli/index.ts <command> [options]

Or using the npm scripts: npm run cli -- <command>

CommandDescription
validate Validate all YAML source files and report errors. Does not write anything. Pass --strict to treat warnings as errors.
migrate Show a dry-run preview of which files need schema migration and what changes would be made.
migrate --rewrite Actually rewrite source YAML files to the current schema version on disk. Only files that differ are written.
rebuild Full pipeline: validate, run rule engine, drop and rebuild SQLite projection.
export Validate, generate forms, then write TEMP/ artifacts (JSON, LaTeX macros, appendix). Does not edit the grammar repo.

Web App

Start the local editing app:

npm run dev:app
# or
npx tsx src/app/server.ts

Then open http://localhost:3737.

App features

  • Browse and search all lexemes and morphemes in the sidebar
  • Filter by category (noun, verb, modifier, function-word) and entry kind
  • View entry details: core info, generated paradigm forms, usage examples
  • Identify manual overrides (marked with ⚡)
  • Rebuild the SQLite projection with the Rebuild DB button
  • Export grammar artifacts to TEMP/ with the Export button

Grammar Repo Sync

The lexicon platform generates artifacts for the grammar repo but never edits the grammar repo directly. All generated files go into TEMP/ inside the lexicon repo:

TEMP/
├── lexicon-export.json      ← full normalized JSON export
├── iridian-lexicon.tex      ← \Lex / \Form / \Morph macro table
├── appendix-lexicon.tex     ← auto-generated paradigm appendix
└── sync-manifest.json       ← metadata about this export run

The grammar repo's build process should copy these files from TEMP/ at compile time (e.g. via a Makefile rule or shell script).

LaTeX Macros

The generated iridian-lexicon.tex file provides three public macros:

MacroWhat it produces
\Lex{key}The display lemma of the lexeme with key key
\Form{key}{slot}The surface form for slot slot of lexeme key
\Morph{key}The display form of morpheme key

Unknown keys or slots emit an \errmessage that fails the LaTeX build loudly, preventing silent data drift between the lexicon and grammar.

% In the grammar preamble:
\input{../iridian-lexicon/TEMP/iridian-lexicon}

% In the body:
The nominative singular of \textit{\Lex{ara}} is \textit{\Form{ara}{NOM.SG}}.

Schema Migration

Two version numbers are tracked independently:

  • source_schema_version — the YAML file format version. Stored in every YAML file as schema_version.
  • db_schema_version — the SQLite projection schema version. Stored in the db_meta table.

When db_schema_version changes, a full SQLite rebuild is required. When source_schema_version changes, migration transforms are applied in-memory during the pipeline. To rewrite source files on disk, run:

npx tsx src/cli/index.ts migrate --rewrite
Warning
migrate --rewrite edits YAML files in place. Commit your work before running this command.

Keys & IDs

Every entry has two identifiers:

  • id — machine identity. A stable, lowercase alphanumeric string (e.g. "ara-001"). Never change this after creation. Used for provenance tracking.
  • key — human-readable stable reference used in YAML cross-references, the app UI, and LaTeX macros. Must be unique across all entries. URL-safe characters only: a-zA-Z0-9 . _ : -.
Naming convention
Lexeme keys are typically the lemma form or a short label: ara, keth-run. Morpheme keys match their Leipzig abbreviation: NOM.SG, PST, AGT. Rule keys use kebab-case: noun-class-a, strong-verb-ii.

Authoring Workflow

Adding a new lexeme

  1. Create a new YAML file in the appropriate lexemes/ subdirectory.
  2. Fill in all required fields. Choose an unused id and key.
  3. Run lexicon validate to check for errors.
  4. Run lexicon rebuild to update SQLite and verify forms are generated correctly.
  5. Inspect the paradigm in the web app under Forms.
  6. Commit the YAML file.

Editing an inflection rule

  1. Edit the rule YAML in rules/inflection/.
  2. Run lexicon rebuild. The rule engine regenerates forms for all affected lexemes.
  3. Check the app to confirm forms are as expected.

Exporting for the grammar repo

  1. Run lexicon export (or click Export in the app).
  2. Review TEMP/sync-manifest.json for the count of exported entries.
  3. Copy TEMP/iridian-lexicon.tex and TEMP/appendix-lexicon.tex into the grammar repo's designated input directory.
  4. Rebuild the grammar PDF to verify no unknown key errors.
